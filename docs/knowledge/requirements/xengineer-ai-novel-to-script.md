## 题目

第三批次第三题：AI 小说转剧本工具。

## 官方需求整理

很多小说作者希望把自己的作品改编成剧本。项目需要开发一款 AI 辅助剧本创作工具，用来降低改编门槛，并提升从小说到剧本初稿的效率。

必须支持的核心能力：

- 输入 3 个章节以上的小说文本。
- 自动转换为结构化剧本。
- 输出格式为 YAML。
- 产物应是可编辑、可继续打磨的剧本初稿。
- 需要额外提交一篇文档，定义剧本的 YAML Schema。
- YAML Schema 文档必须说明 Schema 的设计原因。

## 工程解读

这些不是官方额外要求，而是为了后续工程搭建对官方需求的落地解释：

- “3 个章节以上”意味着导入、解析、切分和生成链路不能只按单章 demo 设计；它是能力证明，不是禁止用户转换单章或两章文本。
- “结构化剧本”意味着输出不能只是自然语言剧本文本，应有稳定字段、层级和可校验结构。
- “YAML 格式”意味着需要考虑人类可编辑性、字段命名、数组顺序和 schema 校验体验。
- “剧本初稿”意味着工具目标不是替作者定稿，而是把小说情节、场景、人物、对白和动作整理成可二次创作的草稿。
- “Schema 设计原因”意味着不能只给字段表，还要说明为什么这样建模、如何支持后续编辑和扩展。

## 最小验收面

为了贴合题意，后续 MVP 至少应覆盖：

- 小说文本导入：支持一次输入或上传包含 3 个以上章节的文本。
- 章节识别：能识别章节边界，并把章节作为生成上下文。
- 改编生成：能把章节内容转成剧本结构，而不是单纯摘要。
- YAML 导出：生成可复制或下载的 YAML。
- Schema 文档：在 `docs/knowledge/schema/script-yaml-schema.md` 中定义字段和设计原因。
- README：说明如何运行、如何输入小说、如何查看 YAML 输出、demo 视频链接和依赖说明。

## 当前风险

- 官方评分强调作品完整度、创新性和交互流畅度，单纯命令行转换可能表达力不足。
- 提交规范要求持续 PR 和 commit，工程搭建后需要按小功能拆 PR。
- 所有 commit 时间戳必须落在第三批次窗口内，历史代码和过往片段需要谨慎处理并注明来源。
