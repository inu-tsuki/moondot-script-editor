# XEngineer 评审与作品提交规范

> 最近更新：2026-06-05  
> 来源：七牛云 x XEngineer 暑期实训营页面 `https://hr.qiniu.com/#review-rules`，以及该页面前端 bundle 中展开后的“作品提交规范”。  
> 适用范围：第三批次第三题“AI 小说转剧本工具”。

## 评审规则

官方按三部分综合评审：

- 作品完整度与创新性：40%。关注产品设计是否合理、功能是否完整、交互是否流畅，以及是否有新颖创意。
- 开发过程与质量：40%。关注架构清晰度、代码健壮度、逻辑、规范、可读性、PR 数量与质量，以及 commit 分布是否合理。
- 演示与表达：20%。关注 demo 视频是否清晰完整，是否能表达作品功能和效果。

## 必交材料

- 公开可访问的 GitHub 或 Gitee 仓库。提交截止后应确保评委可以访问。
- demo 视频。
- README 文档。

## 作品有效性约束

- 作品必须符合所选议题方向。
- 作品必须自主完成，不得抄袭。
- 从议题发布开始，应保持持续 PR 记录和 commit 提交。
- 仅在最后一天一次性导入所有代码，会被视为无效作品。
- commit 时间戳必须落在所选批次开始和截止时间内。
- 第三批次时间按页面标注管理为 `2026-06-05 00:00` 至 `2026-06-07 23:59`（Asia/Shanghai）。

其他会导致无效的风险：

- PR 描述为空，或与实际代码变更明显不一致。
- 使用第三方库或框架，但 README 没有列明依赖，也没有说明原创功能部分。
- 复用自己过去的代码片段，但 PR 描述没有注明来源。

## PR 规范

- 新功能应基于 PR 添加。
- 每个 PR 只做一件事。
- 鼓励小粒度 PR；大功能应拆成多个独立 PR 分步提交。
- PR 标题需要一句话说明新增或修改内容。
- PR 描述需要包含功能作用、使用方式、技术选型或核心实现逻辑、测试方式。
- PR 合并后，主分支必须保持可运行，评委在任意时间查看都应能复现演示效果。

## Demo 视频要求

- 需要用声音完整讲解并展示主要功能和效果。
- 需要覆盖核心模块。
- 视频应上传到可访问的外部平台，例如 Bilibili 或云盘。
- 视频链接需要放到代码仓库 README 中，并确保可播放。

## README 必备信息

官方要求 README 文档，并且提交规范里把 README 与依赖披露绑定。为降低无效风险，README 至少应包含：

- 项目简介和所选议题。
- 本地启动或部署方式。
- 主要功能和使用流程。
- demo 视频链接。
- 第三方库、框架和外部服务依赖。
- 原创功能范围，以及复用代码或参考材料说明。
- YAML Schema 文档链接。

## 仓库与团队要求

- 多人组队时，每队提交一个代码仓库地址。
- 队员应使用各自账号提交 commit。
- PR 描述应清晰呈现各自分工。
- 如果项目包含多个独立模块，应放在同一仓库的不同子目录中，例如 `/frontend` 与 `/backend`。
- 代码仓库需要在开题后创建。

## 提交流程约束

- 页面提示：所选批次议题发布后 24 小时内提交代码仓库地址。
- 作品提交表单要求选择议题，并填写作品仓库链接。
- 仓库链接必须可访问，后续评审以该链接为准。
- 报名或提交后需要关注官方社群，后续批次、议题发布和提交提醒会在群内同步。

## 知识产权与学术诚信

- 除非另有说明，作品知识产权归提交队伍或个人所有。
- 不得侵犯他人知识产权或其他权益。
- 不得未经相关单位许可公开涉及该单位知识产权的作品或软件。
- 如造成不良后果，相关法律责任由提交人自行承担，七牛云不承担相关法律责任。
- 若确认存在代码抄袭、技术抄袭等学术不端行为，或代码重复率在 50% 以上，将取消路演资格，并列入招聘黑名单。

## 提交前检查

- 仓库已公开或评委可访问。
- README 已覆盖运行方式、依赖、原创范围、demo 视频链接和 YAML Schema 文档链接。
- demo 视频可播放，并包含声音讲解。
- PR 记录连续，且每个 PR 描述完整。
- commit 时间均落在第三批次窗口内。
- 主分支可运行，能复现 demo。
