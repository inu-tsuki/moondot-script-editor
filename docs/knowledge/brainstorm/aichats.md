怎么从零开始创作剧本？

### 🤖 Assistant

从零开始创作一个剧本，就像是亲手搭建一座复杂的游乐园——既需要宏大的想象力，也需要精确的工程学。
不需要一开始就字斟句酌地写台词，编剧的核心逻辑是“由粗到细，由内而外”。以下是为你整理的剧本创作“六步法”：

## 核心六步法

### 1. 核心概念（Logline / 一句话故事）
这是你故事的种子。用一句话概括：**谁，在什么背景下，面临什么危机，必须达成什么目标？**
公式： 当 [突发事件] 发生时，一个 [主角特征的主角] 必须去 [核心任务]，否则就会 [灾难性后果]。
- *例：当一颗陨石即将撞击地球时，一个患有恐高症的钻井工人必须进入太空炸毁陨石，否则人类将会灭绝。*

### 2. 人物设定（Character）
人物是故事的引擎。写剧本不是写完美的圣人，而是写**有缺陷、有欲望的凡人**。你需要明确三件事：

- **欲望（Want）：** 他表面上想要达成的具体目标（如：赚大钱）。
- **需求（Need）：** 他内心深处真正缺失、需要成长的地方（如：学会爱与陪伴）。*欲望和需求往往是冲突的。*
- **软肋/缺陷（Flaw）：** 他的致命弱点。这会让观众产生共鸣，也是阻碍他达成目标的内在原因。

### 3. 大纲设计（Structure）
传统的电影或故事多采用经典的**三幕式结构**。你可以把故事想象成一条抛物线：

- **第一幕：建置（占25%）**
  - **常规世界：** 展现主角的日常生活与困境。
  - **催化剂（触发事件）：** 发生了一件事，打破了生活的平衡。
  - **跨越门槛：** 主角被迫或主动选择踏上一条无法回头的旅程（进入第二幕）。
- **第二幕：对抗（占50%）**
  - **中点：** 发生一个重大转折，要么获得阶段性胜利，要么遭遇重大挫败，游戏规则改变。
  - **灵魂黑夜：** 在第二幕结尾，主角遭遇最大危机，似乎一切都搞砸了，陷入绝望。
- **第三幕：结局（占25%）**
  - **高潮决战：** 主角带着成长，与反派或核心困难进行终极对决。
  - **结局：** 危机解决，建立新秩序，主角完成了心理蜕变。

### 4. 场景大纲（Beat Sheet / 故事板）
不要直接写剧本正文！先写一个**场记表（或者用卡片铺在桌上）**。
把故事拆成一个个具体的场景，每场戏写一两句话：

- *第1场：客厅。张三和李四为了一张过期的彩票大吵大闹。*
- *第2场：雨夜街道。张三愤而出门，意外目击了一场车祸。*

### 5. 撰写正文（Scriptwriting）
终于到了动笔阶段。剧本的格式非常讲究，它是写给导演、演员和摄影师看的“说明书”。标准的剧本由三部分循环组成：
场景行（Scene Heading）格式： 场所（内/外） + 地点 + 时间（日/夜） 例：内. 张三的公寓 - 夜动作描写（Action）只写眼睛能看到、耳朵能听到的东西。不要写心理活动（如：“他心里非常后悔” → 改写为：“他狠狠地锤了一下墙，低下了头”）。对白（Dialogue）角色说的话。尽量简练，避免大段的说教。好台词往往有潜台词（Subtext）——字面是一个意思，眼神和语气是另一个意思。
### 6. 修改、修改、再修改
“优秀的剧本不是写出来的，而是改出来的。” 第一稿通常都是“垃圾”，这很正常。写完第一稿后，放它一个星期不要看。然后再用冷酷的眼光去审视：
- 这场戏对推动剧情有帮助吗？如果没有，删掉。
- 这个角色的台词是不是太像编剧自己在说话了？
- 节奏是不是太拖沓了？

## 给新手的实用避坑指南

- **不要“以话代动”（Show, don't tell）：** 能用动作、眼神、画面表达的，绝不要用台词说出来。
- **制造阻碍：** 不要让主角太轻易得到他想要的。他想开门，钥匙偏偏掉进了下水道；他想表白，情敌刚好推门进来。戏就在冲突中。
- **工欲善其事：** 推荐使用专业的剧本排版软件（如 Fade In、Celtx，或者国内常用的 WriterDuet、Trelby），它们会自动帮你排好标准格式，让你专心于创作。
你脑海中目前是有了一个酷炫的画面、一个有趣的人物，还是已经有一个模糊的故事轮廓了？我们可以从这里开始聊起！

### 👤 User

哪里可以看到实际投入应用的剧本？

### 🤖 Assistant

想要看真正“投入应用”的剧本，最优质的渠道就是去看**工业级的电影/电视剧剧本（Screenplays/Scripts）**。
阅读这些剧本不仅能让你看到标准的排版格式，更能让你学习到编剧是如何用纯粹的视觉和听觉语言（Action & Dialogue）来指导导演和演员演戏的。
以下为你整理了几个国内外最硬核、最正规的剧本资源库：

## 1. 英文原版剧本（好莱坞工业标准）
如果你不排斥英文，好莱坞的剧本资源是最丰富的，因为他们有“颁奖季剧本公开”的传统（每年奥斯卡评选期间，各大片方为了拉票，会官方释出当年大热电影的最终版剧本，称为 **FYC Scripts** - For Your Consideration）。

- **IMSDb (Internet Movie Script Database)**
  - **特点：** 电影界的 IMDb。汇集了无数经典和现代好莱坞电影的剧本。
  - **优势：** 纯网页版，查找极其方便，可以直接在浏览器里阅读，是编剧新手的必备字典。
- **Script Slug**
  - **特点：** 界面极其现代、干净，收集了大量高质量的 PDF 原版电影和美剧剧本。
  - **优势：** 它们提供的是原汁原味的 **PDF 扫描件/打印件**，你可以看到最标准的 Hollywood 排版字体（Courier 12pt）和行距格式。
- **Go Into The Story**
  - **特点：** 由黑名单（The Black List，好莱坞著名的未制作优秀剧本盲评机构）的官方博客运营。
  - **优势：** 里面有一个合集，合法免费提供了数百个好莱坞大片（如《盗梦空间》、《疯狂的麦克斯4》、《社交网络》等）的官方 PDF 剧本供下载。

## 2. 中文影视剧本渠道
中文剧本因为版权和行业习惯，公开的工业级 PDF 相对较少，但依然可以通过以下渠道找到非常扎实的学习材料：

- **专业影视/编剧垂直媒体与社群**
  - **新片场 / 拍电影网 / 后浪电影：** 这些公众号或网站经常会分享一些国内优秀电影、短片的剧本拆解，或者限时分享官方授权的电子版剧本（例如《流浪地球》、《流浪地球2》的官方电影剧本集都有实体书出版，网上也有电子版节选）。
  - **专业编剧App（如“小黑盒编剧”/“剧本写生”）的社区：** 很多国内的剧本创作软件内置了经典剧本库，供用户参考格式。
- **实体出版的“电影剧本集”**
  - 如果你想系统学习，非常推荐去书店或电商平台购买**著名导演的电影剧本集**。例如：芦苇的《电影编剧秘密》（内含《霸王别姬》剧本草稿及拆解）、王家卫电影的剧本集、或者是贾樟柯的《贾想》系列。这些实体书保留了最原始的文字风貌。

## 3. 互动叙事 / 游戏 / 动漫剧本
如果你对影视之外的细分领域（如视觉小说、游戏、广播剧等）感兴趣，可以尝试这些渠道：

- **经典游戏文本提取（Script Dumps）：** 很多二次元游戏或文字冒险游戏（Visual Novels）由于其本地化需要，粉丝会把游戏内的文本按表格或剧本格式提取出来（例如在 GitHub 或特定游戏 Wiki 上搜索 "Game Script"）。
- **网络电台/播客剧本：** 很多声音悬疑剧、广播剧的官方社群会公开配音使用的“CV本”或“台本”，这也是非常典型的实用性剧本。

## 💡 怎么读才能榨干剧本的价值？
拿到一份实际应用的剧本后，不要像看小说一样顺着读，试着做这个练习：

1. **拉片对比：** 找一部你非常熟悉的电影，一边播放，一边手里拿着它的剧本。
2. **观察转化：** 看看剧本里写的一句 `“内. 咖啡厅 - 日。两人的气氛降到了冰点。”` 导演在实际拍摄时用了几个镜头？演员用了什么眼神？
3. **学习动作描写：** 重点看编剧是怎么不用“心理活动词汇”去表现人物愤怒、悲伤或紧张的。
如果你想从一个具体的类型（比如科幻、悬疑、或是日常对白极强的剧情片）开始看起，可以告诉我，我能为你推荐一两部该类型最具代表性的标杆剧本！

## Turn 3

### 👤 User

这个固定行字符数，等宽字符，使用ascii风格排版的格式是国内外通用的吗？

### 🤖 Assistant

这是一个非常深刻且触及编剧行业底层的核心问题。
简单直接地回答：**在英文及好莱坞体系中，这是绝对、严格的通用铁律；但在中文影视圈、游戏行业（如文字冒险游戏、动漫画）中，它正在演变成一种“技术规范”或被更灵活的本地化格式所取代。**
我们可以把这种排版拆开来，看看它在国内外不同的“落地”情况：

## 1. 国外（好莱坞及英文体系）：近乎绝对的统治地位
在好莱坞，这种**等宽字体（Monospace）**、固定行字符数、类似 ASCII 风格的排版，有一个神圣不可侵犯的标准：**Courier 12pt（12号库里尔体）**。
为什么他们到今天还死守这个“落后”的打字机时代产物？原因极其工业化：

- **一页一分钟（One Page Per Minute）：** 因为 Courier 是等宽字体，无论你写字母 `i` 还是 `w`，占用的横向空间完全一样。经过长期的工业验证，在标准的页边距下，**标准格式的一页纸剧本，拍成电影刚好等于 1 分钟的屏幕时间**。
- **制片预算与排期工具：**
好莱坞的制片人和副导演在拿到剧本后，会直接根据页数来估算拍摄天数和成本。如果一页纸纸有 $1/8$ 是空白，他们甚至会用八分之一页（$1/8\text{ page}$）来作为基础排期单位。如果你换了别的非等宽字体（比如 Times New Roman），这个公式就彻底崩塌了。
所以在国外，不用这个格式的剧本，**会被软件自动拦截或被好莱坞经纪人直接扔进垃圾桶**。

## 2. 国内（中文影视圈）：从“百花齐放”走向“好莱坞标准规范”
中文由于汉字是**天然的方块字（本身就是等宽的）**，且不使用英文字母，因此国内的剧本发展轨迹有些不同：

### 过去与现状：三足鼎立

1. **好莱坞标准式（逐渐成为主流）：**
现在越来越多国内受过系统训练的年轻编剧、跨国合拍片项目，开始强制使用 Final Draft 或 Fade In 等专业软件。它们输出的中文剧本保留了外文剧本的错落感（动作居左，角色名和对白深深缩进，在中间形成一个窄列）。虽然汉字不如英文那样完美契合“一页一分钟”，但它具备极高的**可读性**和**层次感**。
2. **传统中式/大陆式剧本：**
在国内的老一辈编剧或电视剧圈里，更常见的是一种接近普通文档的排版：`张三：（愤怒地）你到底想怎么样？``李四坐在沙发上，冷冷地看着他。`
这种排版不进行中间缩进，对白和动作混合在一起。虽然省纸，但在工业化流程中（如演员围读、导演现场找戏）不如 ASCII 风格的中间缩进版好用。
3. **港台式（格子本/直排）：**
早期的港台剧本受日本影响或传统习惯，有的甚至使用竖排或特定的表格形式，现在也逐渐向国际标准靠拢。

## 3. 游戏、视觉小说（VN）与动漫行业：演变为 DSL 与标记文本
如果你关注的是游戏剧本（尤其是二次元、文字冒险游戏、视觉小说）或动漫分镜，格式会发生更有趣的演变：
由于游戏剧本最终是要输入到代码引擎（如 Ren'Py、Unity、甚至是自研的文本解析 DSL）中运行的，它往往会和自动化脚本融合，呈现出极其硬核的 **ASCII/代码风格**：

```plaintext
[bg name="sunset_beach" fade=true]
[play_se name="wave_01" loop=true]

# 角色名和对白直接作为解析单元
【主角】
“……如果那天我没有松开手，结局会不会不一样？”

[anim name="heroine" type="shake"]
【女主角】（眼角泛泪）
“笨蛋，现在说这些还有什么用……”

```
在这个领域，比起传统的纸张页数，开发者更关心**文本的行数、变量的调用和标签（Tags）的匹配**。

## 总结：新手应该怎么做？

1. **写影视/短片剧本：** 强烈建议直接采用**好莱坞式的等宽+缩进排版**。不用自己手动敲空格，用专业的编剧软件（如 WriterDuet、Trelby、小黑盒编剧）它们会自动帮你把格式切成那个经典的 ASCII/打字机风格。这不仅能让你显得专业，而且阅读体验极佳。
2. **写游戏/互动叙事：** 格式可以更灵活，通常以“角色标签 + 对白 + 括号内动作标记”为核心，甚至可以根据自己使用的引擎自定义一套纯文本（Plain Text）规范。
你是打算写更偏向大银幕的传统影视剧本，还是带有逻辑和技术标签的游戏/互动类剧本？

## Turn 4

### 👤 User

我正在尝试做一款现代化剧本编辑器，我可以往哪个方向调研？

### 🤖 Assistant

要做一款**现代化的剧本编辑器**，这是一个非常硬核且极具价值的方向。目前的编剧软件（如 Final Draft）大多像二十年前的 Word，虽然符合工业标准，但在现代协作、工程化管理和智能辅助方面已经严重落后。
既然你有心挑战这个领域，可以将调研方向分为**底层语法与解析、编辑器核心生态、现代化功能拓展**三个维度：

## 一、 底层标准与技术选型调研
不要去发明一种不兼容工业标准的存储格式（比如复杂的私有 JSON 或 XML）。现代文本编辑器的核心趋势是“纯文本存储（Plain Text），富文本渲染（WYSIWYG/实时预览）”。

### 1. 语法标准：深入研究 Fountain 协议

- **什么是 Fountain：** 它是编剧界的 Markdown（基于 John Gruber 的理念设计）。它允许编剧用最自然的纯文本写剧本（如：全大写代表角色，`INT.` 开头代表场景），然后通过解析器自动渲染成好莱坞标准排版。
- **调研重点：**
  - 官方规范（`fountain.io`），学习它是如何通过简单的换行和正则来区分“动作（Action）”、“对白（Dialogue）”和“括号（Parenthetical）”的。
  - 开源解析器（如 `screenplay-tools` 或各大语言的 `fountain-js`）。

### 2. 语法高亮与结构化：Tree-sitter & LSP
传统的正则高亮在面对数万行的剧本时会卡顿，且无法理解上下文。

- **Tree-sitter：** 调研如何为剧本语法（无论是 Fountain 还是你自己为游戏设计的 DSL）编写一个 Tree-sitter 解析器。它能将剧本实时解析成**增量语法树（AST）**。
- **LSP（Language Server Protocol）：** 现代化编辑器（如 VS Code、Zed、Neovim）强大的核心在于 LSP。你可以把剧本的“角色名自动补全”、“场景跳转（Go to Definition）”、“冲突检查”包装成一个剧本的 Language Server，这样你的编辑器底层会极其稳固和专业。

## 二、 现代化编辑器架构（前端/客户端）
如果选择 Web/跨平台技术栈，可以重点调研以下两家独大的编辑器框架：

### 1. Lexical (Meta 出品) vs ProseMirror

- **不要使用原生的 contentEditable**，那是一场灾难。
- **Lexical / ProseMirror：** 它们是基于状态（State-driven）的富文本编辑器内核。你可以利用它们构建一个“所见即所得”的剧本环境：用户输入纯文本，但编辑器实时将其排版成带缩进、等宽字体的标准剧本样式。
- **调研对象：** 看看 **Arc Studio** 或 **Highland 3**。它们完美做到了“写作时像 Markdown 一样流畅，导出的 PDF 又完全符合工业打字机规范”。

## 三、 功能破局点：现代编剧的痛点方向
这是你的编辑器脱颖而出的地方。传统软件只管“写”，现代编辑器应该管“编”。

### 1. 结构化大纲与故事板（Outlining & Beats）

- 编剧在动笔前需要铺卡片。调研编辑器如何实现“双向绑定”：左边是无限画布的卡片墙（每一张卡片代表一个场景），右边是剧本正文。拖动卡片，右边的剧本章节自动调换顺序。

### 2. 角色与资产字典（Asset Management）

- 当编剧在剧本中打出 `@张三` 时，编辑器应该能自动关联出张三的人物小传、出场场次统计、台词占比。
- 调研方向：Notion 的行内提及（Mention）与关系型数据库（Relation）在剧本场景下的本地化实现。

### 3. 全自动化工业制片导出（Production Breakdown）

- 副导演和制片人拿到剧本后，要人工数出这个角色有多少场戏、在哪里拍、是白天还是黑夜、需要什么道具。
- 你的编辑器如果能通过 AST 自动分析剧本，**一键生成“制片拆分配表（Breakdown Sheet）”**，那将是降维打击。

### 4. 协同与版本控制（Git-like for Writers）

- 影视行业改本子是家常便饭。传统软件用“红线圈阅”极其混乱。

## 一、 中文标准排版：双轨制支持（Fountain 语法的本地化）
中文影视圈目前存在“好莱坞工业派”和“传统大陆派”两种习惯。你的编辑器应该用**同一套底层纯文本语法（如 Fountain 变体），通过 CSS 切换不同的渲染视图**。

### 1. 视图 A：国际工业流（中间对齐缩进）

- **角色名（Character）：** 居中（或居左大幅缩进，通常占总宽度的 $40\%$ 位置）。
- **对白（Dialogue）：** 在中间形成一个垂直的窄列（左右各缩进十几字符）。

### 2. 视图 B：传统中式流（全两端对齐）
国内电视剧、网剧、小品小剧场更习惯这种“省纸”格式。

## 二、 中文分词与 AST 解析（Tree-sitter）
由于中文**不使用空格分隔单词**，传统的基于空格和西文符号判断的 Fountain 解析器在处理中文时会彻底失效。

### 1. 场景行（Scene Heading）的特殊解析
英文通过 `INT.` 或 `EXT.` 识别场景。中文对应的则是 `内.`、`外.`、`内外.`。

- **解析规则：** 需要通过 Tree-sitter 或正则精确匹配：`^(内|外|内外)\.?\s*([^-\n]+)\s*-\s*(日|夜|黄昏|黎明)`。
- **现代化优化：** 自动为中文场景提供快捷键或自动补全。输入 `n` 自动带出 `内.`，输入 `w` 自动带出 `外.`。

### 2. 括号（Parenthetical）的本地化
英文剧本用 `( )` 包裹角色的神态动作。中文编剧喜欢用全角括号 `（）`，甚至有人喜欢用 `[ ]` 或 `【 】`。

- **调研方向：** 你的编辑器必须支持全角括号的语义识别。当检测到 `【生气地】` 出现在角色名和对白之间时，AST 必须将其解析为 `Parenthetical` 节点，而不是普通对白，从而在排版时给予它特有的缩进或变色。

  - 通常情况下，华语影视剧的语速约为 200~250 字/分钟，动作描写则需要根据句式复杂度转化为导演调度时间。

中文是一门极其含蓄、注重潜台词的语言（例如：“你看着办吧”字面是放任，潜台词往往是极度愤怒）。

既然你要做现代编辑器，绝对**不能**用一整串纯 String 来表达剧本。必须引入**结构化的块级数据模型（Block-based Definition）**。结合中文和英文的混排需求，可以设计如下的 TypeScript 类型定义：

```typescript
// 1. 剧本节点的底层类型
export type ScriptBlockType = 
  | 'TITLE'          // 剧本标题/分区
  | 'SCENE_HEADING'  // 场景行 (INT/EXT)
  | 'ACTION'         // 动作描写
  | 'CHARACTER'      // 角色名
  | 'DIALOGUE'       // 对白
  | 'PARENTHETICAL'  // 括号神态/技术标记 (如 con't, TV)
  | 'NOTE'           // 制作批注/Memo
  | 'TRANSITION';    // 切入切出 (如 FADE IN:)

// 2. 结构化的块级实体
export interface ScriptBlock {
  id: string;                  // 唯一标识，用于 CRDT (Yjs) 协同和 DOM 渲染映射
  type: ScriptBlockType;
  text: string;                // 实际文本内容
  
  // 工业元数据 (Meta Data)
  meta?: {
    sceneNumber?: string;      // 场景号，如 "A1", "B4"
    sceneSetting?: 'INT' | 'EXT' | 'INT_EXT'; // 内外景
    location?: string;         // 地点：如 "CAVES"
    timeOfDay?: string;        // 时间：如 "NIGHT", "DIURNAL"
    characterId?: string;      // 如果是角色/对白块，关联资产库 ID
    parentheticalText?: string;// 括号内的附加文本
    updatedAt?: string;        // 对应剧本中的修改日期 "12/7/65"
  };
}

// 3. 剧本树状大纲结构（用于双向绑定故事板）
export interface SceneNode {
  id: string;
  title: string;               // 场景标题
  sceneNumber: string;
  blocks: ScriptBlock[];       // 该场景包含的所有文本块
  estimatedDuration: number;   // 智能预估时长（秒）
}

export interface ScreenplayDoc {
  documentId: string;
  title: string;
  metaData: Record<string, any>;
  scenes: SceneNode[];         // 场景流
}

```

### 四、 架构破局：用 Fountain 还是 Fountain + 自定义数据结构协同？
**结论：必须采用“Fountain 作为物理存储/传输协议 + 结构化树（AST/JSON）作为内存状态”的协同模式。**
如果只用其中一个，你的编辑器会走向两个极端的灾难：

- **只用 Fountain 纯文本：** 当你需要做“拖拽场景卡片换位”、“点击角色查看所有出场统计”、“一键导出游戏脚本”时，每一次操作都需要全局正则解析几万行的文本，性能会彻底卡死，且极难处理协同。
- **只用私有 JSON：** 用户无法用任何其他软件打开你的剧本，失去了纯文本的生态优势，且无法做优雅的文本 Diff。

#### 完美的协同架构应该如下运行：

```plaintext
NARRATOR (con't)
Those who had begun the experiment so long ago had not been men.

```

#### 这样设计的好处：
编剧在打字时，体验到的是像 Markdown 一样极其流畅、没有任何弹窗干扰的**纯文本输入**；但你的编辑器后台拿到的是**高度结构化的工业工程数据**，可以玩转任何现代化的自动化制片功能。

将 **Fountain 配合 YAML** 是一个在现代剧本编辑器设计中**极其天才且非常成熟的解法**。
事实上，这种模式在现代纯文本生产力工具中（如 Obsidian 的 Front Matter、Hugo/Hexo 等静态网站生成器的 Markdown+YAML 头部）已经被验证得非常彻底。它可以完美解决“Fountain 负责线性文本叙事，YAML 负责非线性结构化元数据（Metadata）”的解耦需求。

## 一、 它们该如何优雅地“合体”？
参考现代文档工程的标准，最优雅的配合方式是 **混合文本（Hybrid Text）** 模式。将 YAML 作为剧本文件的 **Front Matter（前置元导）** 或 **Scene Matter（场景元导）**。

### 1. 全局配置（文件头部 Front Matter）
在剧本文件的最顶部，用三个连字符 `---` 包裹 YAML 数据。这里存放整个剧本的资产、全局变量、甚至多分支剧情的路由表。

```yaml
---
title: 2001: A Space Odyssey
writers:
  - Stanley Kubrick
  - Arthur C. Clarke
version: "Production Draft 1965"
characters:
  - id: ch_moonwatcher
    name: MOONWATCHER
    role: Protagonist
    traits: [Hominid, Leader, Primitive]
  - id: ch_hal9000
    name: HAL 9000
    role: Antagonist
---

# 这里开始是标准的 Fountain 文本

TITLE PART I AFRICA 3,000,000 YEARS AGO

EXT. AFRICAN DRYLANDS - DROUGHT

The remorseless drought had lasted now for ten million years...

```

### 2. 局部配置（场景级 Scene Matter，针对游戏/互动叙事扩展）
原生 Fountain 用 `#` 表示章节，用 `.` 表示场景。如果你在写游戏剧本、多结局短剧，可以在场景下方插入局部 YAML，用来注入**代码逻辑、游戏资产引用或条件分支**。

```plaintext
# 章节：太空站遭遇

. INT. SPACE STATION RECEPTION AREA - DAY

RECEPTIONIST at desk. MILLER enters, hurrying.

```

## 二、 为什么说 Fountain + YAML 是绝配？（核心优势）

### 1. 完美解决 Fountain 的“非线性短板”
Fountain 擅长处理 **“时间线纯线性的影视剧本”**（从第一页顺着演到第一百页）。但是，现代剧本（尤其是游戏 VN、互动短剧、带宏大资产的科幻片）是**网状的、非线性的**。

- **Fountain 管“皮肉”：** 场景行、动作、对白、神态。
- **YAML 管“骨骼”：** 道具 ID、角色属性变化、数值埋点、逻辑路由。两者各司其职。

### 2. 人类可读性（Human-Readable）与 Git 友好
比起把元数据强行揉成复杂的 JSON 或者像 XML 那样充满标签，YAML 的缩进格式极其干净。对于写剧本的程序员或技术编剧，直接用 Neovim 或任何文本编辑器打开这个文件，一眼就能看懂并手动修改，且**在 Git 中做 Diff 和版本比对时清晰无比**。

### 3. 解析器的生态借力
你不需要自己写一套复杂的元数据解析器。各种语言都有现成、极度成熟的 `js-yaml`、`yaml-cpp` 或 Rust 的 `serde_yaml`。你只需要在编辑器底层写一个简单的 **Splitter（分流器）**：

1. 正则匹配头部的 `---` 或特定的注释块。
2. 上半部分丢给 YAML 解析器 $\rightarrow$ 转化为结构化的资产对象。
3. 下半部分丢给 Fountain 解析器 / Tree-sitter $\rightarrow$ 转化为剧本 AST。

## 三、 编辑器底层的核心数据流设计
当你把 Fountain + YAML 组合在一起后，你的现代化编辑器在读写文件时，数据流应该长这样：

### 1. 读盘（Load/Parse）

1. 用户打开一个 `.fountain` 或 `.vn` 文件。
2. 编辑器后台通过分流器，剥离出 `YAML_STRING` 和 `FOUNTAIN_STRING`。
3. YAML 转化为 `AssetManagerStore`（用于在侧边栏显示角色头像、道具列表）。
4. Fountain 转化为我们上一次讨论的 `ScriptBlock[]`（用于富文本编辑器的渲染）。
5. **双向锚定：** 检查剧本正文中出现的 `CHARACTER` 块（如 `MOONWATCHER`），如果它在 YAML 的 `characters` 列表中能找到，就在编辑器里将该文本高亮，并允许用户悬浮查看 YAML 里定义的 `traits`。

### 2. 落盘（Save/Serialize）
用户在你的 UI 界面里操作（比如在可视化画布里拖拽修改了剧情走向的分支，或者在角色面板里改了名字）。

1. 编辑器将角色面板的修改序列化为 YAML 字符串。
2. 将正文的文本块序列化为 Fountain 字符串。
3. 用 `---` 拼接，写入本地文件。

## ⚠️ 落地时需要注意的一个“深水坑”
**标点符号与特殊字符的冲突。**
中文剧本里有大量的冒号（`：`）、破折号（`——`）、感叹号（`！`）。

- 在 Fountain 中，由于它是普通文本，这些标点没有任何问题。
- 如果你在 YAML 的字段里也写了中文对白（比如做游戏的剧情选项分支：`- choice: "等等！你别走："`），**YAML 解析器可能会因为中文的冒号或特殊标点产生语意解析错误**。
**解决方案：** 在编辑器序列化生成 YAML 部分时，**强制给所有的字符串值加上双引号 ""**（例如 `- choice: "等等！你别走："`）。这能保证你的 YAML 解析器绝对安全，不会因为中文编剧天马行空的标点符号而崩溃。
这个方案非常适合你想要打造的“现代化、本地优先（Local-first）、工程化”的剧本编辑器，完全具备对抗老牌工业软件的底层实力！

  let output = `---\n${yaml.stringify(project.manifest)}---\n\n`; // 塞入 YAML
  
  for (const node of getLinearNodes(project.graph)) {
    output += `\n.${node.title}\n\n`; // 场景行
    for (const block of node.blocks) {
      if (block.type === 'CHARACTER') output += block.text.toUpperCase() + '\n';
      if (block.type === 'DIALOGUE') output += block.text + '\n\n';

    }
  }
  return output;
}

#### 2. 编译为 Ren'Py 脚本（视觉小说游戏）

```typescript
function compileToRenpy(project: ScreenplayProject): string {
  let output = "";
  // 1. 将 Manifest 中的角色声明为 Ren'Py 变量
  for (const char of project.manifest.characters) {
    output += `define ${char.alias || char.id} = Character("${char.name}")\n`;
  }
  
  // 2. 遍历图节点，生成 label 和 menu
  for (const [id, node] of Object.entries(project.graph.nodes)) {
    output += `\nlabel ${id}:\n`;
    if (node.meta.background) output += `    scene bg ${node.meta.background}\n`;
    
    for (const block of node.blocks) {
      if (block.type === 'DIALOGUE') {
        output += `    ${block.binding?.characterId} "${block.text}"\n`;
      }
    }
    
    // 3. 处理分支选项
    if (node.routing.nextType === 'CHOICE') {
      output += `    menu:\n`;
      for (const choice of node.routing.choices) {
        output += `        "${choice.text}":\n            jump ${choice.targetNodeId}\n`;
      }
    }
  }
  return output;
}

```

## 这个数据结构的降维打击优势

1. **游戏分支与影视章节的统一：**
在影视模式下，`routing.nextType` 永远是 `LINEAR`。用户的视图是一个没有分支的线性流，`SceneNode` 的数组顺序就是剧本的页码顺序。而在游戏模式下，编辑器切换为“网络图（Node Graph）”画布，`routing.choices` 变成了连线，用户在画布上连线，就是在修改 `targetNodeId`。
2. **对白资产零耦合：**
如果剧本里的角色需要改名字（比如把 `MOONWATCHER` 改成 `望月者`），你不需要全局去替换成千上万行文本，你只需要在最上层的 `GlobalManifest.characters` 里把 `name` 改掉。在渲染层或导出层，系统会自动动态替换。这对于动辄几十万字、需要反复修改设定的中文大作来说，是真正的工业救星。
